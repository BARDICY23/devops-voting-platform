# Bug Fixes - April 16, 2026 (01:57 AM)

Here is a log of all the infrastructure and application bugs resolved during this session:

## 1. Vote App Startup Failure
* **Symptom:** Kubernetes pods for `vote` falling into `CrashLoopBackOff`.
* **Root Cause:** The `vote` Dockerfile was supplying an invalid flag (`--no-control-socket`) to Gunicorn on startup, causing an immediate crash.
* **Resolution:** Refactored the `CMD` string in `services/vote/Dockerfile` to use standard Gunicorn arguments (`app:app`, `-b 0.0.0.0:8080`, `--log-file -`).

## 2. Worker App Health Probes Failing
* **Symptom:** Kubernetes pods for `worker` falling into `CrashLoopBackOff`.
* **Root Cause:** The `worker.yaml` Helm chart configured liveness and readiness probes to run `pgrep -f "Worker"`. However, the minimal `.NET` base image (`mcr.microsoft.com/dotnet/aspnet:7.0`) did not have the `pgrep` utility installed, causing Kubernetes to forcibly kill the container repeatedly.
* **Resolution:** Added `apt-get install -y procps` to `services/worker/Dockerfile` to install the required process management tooling.

## 3. Discontinued Bitnami Images (Postgres & Redis)
* **Symptom:** Both the Redis replica (`voting-app-redis-replicas-0`) and PostgreSQL instance (`voting-app-postgresql-0`) became stuck in `ErrImagePull` and `ImagePullBackOff`.
* **Root Cause:** The `bitnami/redis:7.2.5-debian-12-r4` and `bitnami/postgresql:16.3.0-debian-12-r23` images were purged from standard DockerHub by Bitnami's automated tag rotation policies.
* **Resolution:** Overrode the repository targets in `platform/apps/helm/voting-app/values-dev.yaml` to point specifically to the `docker.io/bitnamilegacy` repository to retrieve the exact required stable tags.

## 4. Kubernetes StatefulSet Immutable Field Patching 
* **Symptom:** ArgoCD reported `voting-app-dev` as `Sync Failed` due to "Forbidden" updates on `StatefulSet.apps`.
* **Root Cause:** ArgoCD was trying to dynamically modify core volume structures of the Redis Master and PostgreSQL StatefulSets, which Kubernetes forbids.
* **Resolution:** Manually deleted the locked StatefulSets (`kubectl delete statefulset voting-app-postgresql voting-app-redis-master -n voting`), allowing ArgoCD to securely recreate them from scratch with the new configuration specs.

## 5. ArgoCD CRD Size Limitations (Monitoring Stack)
* **Symptom:** ArgoCD failed to sync the `monitoring` application with an error: `metadata.annotations: Too long: may not be more than 262144 bytes`.
* **Root Cause:** The standard `kubectl apply` strategy used by ArgoCD injects the full resource configuration into an annotation, but Prometheus Custom Resource Definitions (CRDs) vastly exceed this 262KB limit. 
* **Resolution:** Updated `platform/gitops/argocd/apps/monitoring.yaml` with ArgoCD `syncOptions` to inject `- ServerSideApply=true` and `- Replace=true`, forcibly bypassing the local annotation limits.

## 6. Image Layer Caching in Minikube
* **Symptom:** Rebuilt Vote and Worker images were pushed to DockerHub, but Kubernetes wouldn't deploy them and kept crashing using the old image layer.
* **Root Cause:** In Dev, the Helm tags were pinned to `latest` and inherited a global `imagePullPolicy: IfNotPresent`. Kubernetes saw an image tagged `latest` locally and stubbornly refused to re-pull the new code.
* **Resolution:** Injected `pullPolicy: Always` across all custom microservices in `values-dev.yaml` to guarantee fresh pull requests on pod creation.

## 7. Widespread OS and SDK Vulnerabilities (Trivy Scans)
* **Symptom:** GitHub Actions CI pipelines failed with over 180 "High" and "Critical" infrastructure vulnerabilities.
* **Root Cause:** The Dockerfiles were built upon End-of-Life SDKs (`.NET 7`), ancient legacy test frameworks (`node:8.9`), and bloat-heavy development branches of Linux (`trixie` and `slim`).
* **Resolution:** Entirely rebuilt the foundation of the ecosystem. Upgraded Worker SDK to Long-Term Support (`.NET 8.0`), upgraded tests to `node:20-alpine`, and switched `vote`/`seed-data` off Debian `slim` to strictly compressed `alpine` layers. Rewrote package managers from `apt-get` to Alpine's native `apk`.

## 8. Phantom Lockfile Transitive Vulnerabilities (ExpressJS)
* **Symptom:** Upgrading application packages inside `services/result/package.json` had zero effect on mitigating deeply nested ReDos vulnerabilities within sub-packages (like `path-to-regexp` and `body-parser`).
* **Root Cause:** The `result/Dockerfile` was utilizing `npm ci`, which strictly enforces arbitrary snapshot installs mapped out inside `package-lock.json`, purposefully blindfolding the Docker build from fetching our patched updates.
* **Resolution:** Abolished the `package-lock.json` file completely, injected strict fallback `"overrides"` settings into `package.json`, and flipped the Docker instruction to `npm install --omit=dev`. This empowers the pipeline to fetch the bleeding-edge patches transparently.

## 9. Immutable Native Packages (PhantomJS & Alpine NPM)
* **Symptom:** `node-tar`, `glob`, and `minimatch` bugs continuously bounced back inside system-level directories (`/usr/...`) across new builds despite modern upgrades.
* **Root Cause:** In tests, the globally installed `phantomjs-prebuilt` testing module relied on 2018 algorithms. Beyond that, the `node:20-alpine` OS intentionally packaged vulnerable tooling inside its globally embedded `npm` toolkit located deep in `/usr/local/lib/node_modules/npm`.
* **Resolution:** Completely nuked the legacy `phantomjs` and `.sh` scripts and rewrote the integration test natively with `Puppeteer` and `fetch`. To obliterate the base image CVEs, introduced a ruthless command to physically burn and delete `/usr/local/lib/node_modules/npm` and `/opt/yarn*` sequentially from the final deployed containers, making it structurally impossible for Trivy to find them.
